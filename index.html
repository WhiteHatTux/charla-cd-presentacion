Qué es Continuous Deployment?

Va después de Continuous Delivery

Va después de Continuous Integration

Va después de Continuous Build

Es aplicar cualquier cambio en produccion

  • Nuevos features
  • Cambios de configuración
  • Correcciones de errores
  • Experimentos
  • De forma segura y rápida

Qué pasó ahí?

Fases de continuous Deployment

  1. Cambio en el código
  2. Compilación y pruebas unitarias
  3. Pruebas de servicio entero
  4. Instalación en producción
  5. Prueba mínima que la instalación en producción fue exitosa

Por qué hacemos todo esto?

Presunciones - situación actual

  • Instalación es manual y lento
  • Instalación es riesgosa, porque solo se instala cada año
  • Nadie quiere tocar el servidor de producción
  • Equipo de testing asegura calidad manualmente
  • Negocio se mueve más rápido que el Software

Solución

  • Automaticemos
  • Instalar no debería ser algo especial
  • Hagamos rutina de la instalación
  • Escribamos pruebas automáticas
  • Instalemos nuevas funciones lo más rápido posible

Bajar el riesgo de cada instalación

Si instalar es lo más normal del mundo es más probable que nos salga bien

Mejorar el "feedback loop"

El tiempo que tarda un error desde que lo escriba hasta saber que he escrito un error

Mejorar el "Time to market"

Tiempo que necesita un cambio desde su pedido hasta su disponibilidad para el cliente

Mejorar la calidad del código

Instalando las herramientas nos damos cuenta mucho más rápido que hay una falla y encima nos damos cuenta cuando hay una parte del código que no está cubierta por pruebas

Por qué NO hacemos esto?

  • Porque no tenemos suficiente pruebas automáticas
  • Porque no tenemos la mentalidad de desarrollo agil
  • Porque tenemos responsabilidades divididas
  • Porque el software que tenemos no se deja instalar de forma automática

Cómo hay que proceder?

Sistema de control de versiones

Sugerido es git

De hecho no es solo sugerido

Es obligatorio!

Casi siempre va con github/gitlab/bitbucket

Algunos ofrecen repositorios privados or premium para esudiantes

Verificar pruebas del código

  • Sin pruebas no hacemos nada
  • Se puede ejecutar de forma automática?
  • Se puede analizar el resultado de las pruebas?
  • Qué tan grande es la cobertura del código con pruebas?

Usar analysis de codigo estático y dinámico

  • Así entendemos más el código y lo que nos esconde
  • lint
  • checkstyle/pmd/findbugs
  • Sonarqube

Setup Continuous Integration server

  • Que entorno necesita el programa para compilar
    • Script para entorno, no manualmente
  • Crear Jenkinsfile o .travis.yml para setup del entorno y compilación
  • Jenkinsfile or .travis.yml van con el código y permiten evolucionar con él

Aplicar configuration as code

  • Permite versionar y manejar de forma automática todos los cambios
  • y accesos
  • Junto con "configuration as code" va configuration management
  • Por ejemplo consul o etcd
