Les voy a mostrar un ejemplo real de la teoría que les hablo y quiero que reconozcan las siguientes etapas

Qué pasó ahí?

Fases de continuous Deployment

  1. Cambio en el código
  2. Compilación y pruebas unitarias
  3. Comprobar que los cambios a la base de datos funcionan
  4. Pruebas de servicio entero
  5. Instalar en producción
  6. Prueba mínima que la instalación en producción fue exitosa

Por qué hacemos todo esto?

Presunciones - situación actual

  • Instalación es manual y lento
  • Instalación es riesgosa, porque solo se instala cada año
  • Nadie quiere tocar el servidor de producción
  • Equipo de testing asegura calidad manualmente
  • Negocio se mueve más rápido que el Software

Solución

  • Automaticemos
  • Instalar no debería ser algo especial
  • Hagamos rutina de la instalación
  • Escribamos pruebas automáticas
  • Instalemos nuevas funciones lo más rápido posible

Mejorar el "feedback loop"

El tiempo que tarda un error de que lo escriba hasta saber que he escrito un error

Mejorar el "Time to market"

Tiempo que necesita un cambio desde su pedido hasta su disponibilidad para el cliente

Mejorar la calidad del código

Por el simple hecho de saber que cualquier linea de código va directo a producción producimos código de mayor calidad

Por qué NO hacemos esto?

  • Porque no tenemos suficiente pruebas automáticas
  • Porque no tenemos la mentalidad de desarrollo agil
  • Porque tenemos responsabilidades divididas
  • Porque el software que tenemos no se deja instalar de forma automática
De hecho es algo que casi siempre se decide al momento de empezar el proyecto

Es bien complicado aplicarlo cuando un proyecto ya está en marcha

Cómo hay que proceder?

Verificar pruebas del código

  • Sin pruebas no hacemos nada
  • Se puede ejecutar de forma automática?
  • Se puede analizar el resultado de las pruebas?
  • Qué tan grande es la cobertura del código con pruebas?
